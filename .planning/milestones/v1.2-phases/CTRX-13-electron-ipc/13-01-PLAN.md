<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer -> preload | 非可信 UI 输入跨越到受信任 IPC 边界 |
| preload -> main process | 高权限系统操作（文件写入、子进程管理） |
| main process -> local backend | 本地子进程崩溃或配置错误可能影响可用性 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | E | `preload.ts` | mitigate | 仅暴露三通道白名单，禁止透传 `ipcRenderer`（D-05） |
| T-13-02 | I | `store.ts` / logs | mitigate | API Key 脱敏与日志禁明文（D-07） |
| T-13-03 | D | `main.ts` lifecycle | mitigate | 30s 超时 + 3 次退避重试 + 可重试错误弹窗（D-01/D-02/D-04） |
| T-13-04 | T | `save-settings` input | mitigate | main 侧 schema 校验 URL/必填字段（D-06） |
| T-13-05 | R | settings restart flow | mitigate | 保存-重启-健康通过才算成功，失败回滚并记录原因（D-15） |
</threat_model>

<risk_and_rollback>
## Top Risks
- R1: 后端崩溃风暴导致界面不可用。  
  Mitigation: 固定退避（0/2/5s）+ 最大 3 次 + 一键重试（D-02/D-04）。
- R2: IPC 合约漂移导致前后端字段不一致。  
  Mitigation: 单一 `electron/types/ipc.ts` + `window-api.d.ts` 对齐。
- R3: IPC 合约通过但前端体验层未接入。  
  Mitigation: 在 13-02 接入向导/Settings，并复用同一合约类型。

## Rollback Trigger
- 任一条件触发：健康检查超时、连续 3 次崩溃、配置校验失败导致不可恢复。

## Rollback Actions
1. 保留旧配置快照并恢复到 `userData/config.json`。  
2. 使用旧配置重新拉起 Python 子进程并执行健康轮询。  
3. 向用户展示失败原因、日志路径与“重试”入口。  
4. 若仍失败，进入 failed 态并保留重试入口，不放行主流程。
</risk_and_rollback>
