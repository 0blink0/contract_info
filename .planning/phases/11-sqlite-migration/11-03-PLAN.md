def _bundle_base() -> Path:
    """Resolve base directory for bundled read-only assets (templates, dicts, alembic)."""
    if getattr(sys, "frozen", False) and hasattr(sys, "_MEIPASS"):
        return Path(sys._MEIPASS)
    return Path(__file__).resolve().parents[2]

def data_dir() -> Path:
    """Resolve user-writable data directory (uploads, exports, DB file).
    Desktop mode: CTRX_DATA_DIR env var.
    Dev/Docker mode: PROJECT_ROOT (unchanged behaviour).
    """
    raw = os.environ.get("CTRX_DATA_DIR", "").strip()
    if raw:
        p = Path(raw)
        p.mkdir(parents=True, exist_ok=True)
        return p
    return PROJECT_ROOT

def uploads_dir() -> Path:
    return data_dir() / "uploads"

def exports_dir() -> Path:
    return data_dir() / "exports"

def templates_dir() -> Path:
    return _bundle_base() / "templates"   # read-only bundle asset

DICTS_DIR_PATH = _bundle_base() / "dicts"   # module-level, for extract/validate.py

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Electron main → CTRX_DATA_DIR env var | Electron sets CTRX_DATA_DIR before spawning Python; Python reads it. Attacker cannot set env vars for a subprocess they do not control. |
| HTTP input → file path resolution | _resolve_export_path in jobs.py still validates resolved path is within exports_dir(); switching base from PROJECT_ROOT to data_dir() preserves this check. |
| bundle assets → dicts/ | DICTS_DIR now points to _bundle_base() / "dicts"; in dev mode this is PROJECT_ROOT/dicts; in frozen mode it is sys._MEIPASS/dicts — read-only location, no path traversal surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-06 | Elevation of Privilege | CTRX_DATA_DIR set to a system path (e.g. C:\Windows\System32) | accept | CTRX_DATA_DIR is set by the Electron main process (trusted), not from HTTP input. Directory creation via p.mkdir(parents=True, exist_ok=True) may create dirs in unexpected locations if attacker controls env; mitigated because only desktop_main.py sets this var. |
| T-11-07 | Information Disclosure | _resolve_export_path path traversal — still present after change | mitigate | The guard "full.relative_to(root)" in jobs.py _resolve_export_path remains in place; base changes from PROJECT_ROOT to data_dir() but the check is preserved, so traversal outside exports_dir() still raises 404. |
| T-11-08 | Spoofing | _bundle_base() follows sys._MEIPASS which could be set by attacker in test env | accept | sys._MEIPASS is set by PyInstaller at bundle startup, not from HTTP input; test-only monkeypatching is scoped to test process. |
| T-11-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this phase. |
</threat_model>
